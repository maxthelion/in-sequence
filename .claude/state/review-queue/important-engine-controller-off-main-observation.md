# Important: EngineController mutates observed UI state off the main thread

`TickClock.start` invokes its callback on a private dispatch queue (`Sources/Engine/TickClock.swift:40-65`), and `EngineController.start()` forwards that callback straight into `processTick` (`Sources/Engine/EngineController.swift:47-59`). `processTick` then mutates `currentBPM` and `transportPosition` on an `@Observable` object that SwiftUI views are reading (`Sources/Engine/EngineController.swift:175-190`). The checklist explicitly requires the threading contract for UI-facing state to be declared and enforced, but this controller is neither `@MainActor` nor marshaling those mutations back to the main actor. That leaves the transport/status UI relying on cross-thread observation updates and makes data races likely as the engine grows. The controller should either be main-actor isolated with the tick callback hopping back to main for state publication, or split realtime work from a main-thread presentation model.
